# 보관중인 파일 삭제 및 백업 프로세스 개발 (RQ-PR-IT20260018) 상세 변경사항

이 문서는 보관중인 첨부파일 중 등록일자 기준으로 5년 이상된 대상을 선별하여 결재를 진행하고, 결재 승인이 완료된 파일을 시스템 배치(Batch) 작업을 통해 안전하게 백업 경로로 이동시키며, 이동 3개월 후에 영구 삭제하는 프로세스의 신규 추가 및 변경된 정보 상세 보고서입니다.

---

## 1. 데이터베이스 변경 사항 (DDL)

결재 이력 관리를 위한 마스터 테이블 `STOCATCN`을 신규 생성하고, 개별 파일(`STOCATCH`)에서 해당 결재 정보를 직접 참조할 수 있도록 컬럼을 추가합니다.

```sql
-- 1) 신규 결재 관리 테이블 생성 (Oracle 기준)
CREATE TABLE STOCATCN (
    GATE_CD VARCHAR2(10) NOT NULL,
    APP_DOC_NUM VARCHAR2(30) NOT NULL,
    APP_DOC_CNT NUMERIC(6) NOT NULL,
    REG_DATE DATE DEFAULT SYSDATE NOT NULL,
    REG_USER_ID VARCHAR2(50) NOT NULL,
    DEL_FLAG VARCHAR2(1) DEFAULT '0' NOT NULL,
    SIGN_STATUS VARCHAR2(10) NOT NULL, -- P: 결재중(상신), E: 승인(완료), R: 반려, C: 취소/회수
    SIGN_DATE DATE,
    CONSTRAINT PK_STOCATCN PRIMARY KEY (GATE_CD, APP_DOC_NUM, APP_DOC_CNT)
);

COMMENT ON TABLE STOCATCN IS '파일 삭제 결재 관리 마스터 테이블';
COMMENT ON COLUMN STOCATCN.GATE_CD IS '게이트 코드';
COMMENT ON COLUMN STOCATCN.APP_DOC_NUM IS '결재 문서 번호';
COMMENT ON COLUMN STOCATCN.APP_DOC_CNT IS '결재 차수';
COMMENT ON COLUMN STOCATCN.REG_DATE IS '등록 일자';
COMMENT ON COLUMN STOCATCN.REG_USER_ID IS '등록자 ID';
COMMENT ON COLUMN STOCATCN.DEL_FLAG IS '삭제 여부';
COMMENT ON COLUMN STOCATCN.SIGN_STATUS IS '결재 상태 (P:상신, E:승인, R:반려, C:취소)';
COMMENT ON COLUMN STOCATCN.SIGN_DATE IS '결재 승인 일자';

-- 2) 기존 첨부파일 공통 테이블에 결재 정보 컬럼 추가
ALTER TABLE STOCATCH ADD (
    APP_DOC_NUM VARCHAR2(30) NULL,
    APP_DOC_CNT NUMERIC(6) NULL
);

COMMENT ON COLUMN STOCATCH.APP_DOC_NUM IS '삭제 결재 문서 번호';
COMMENT ON COLUMN STOCATCH.APP_DOC_CNT IS '삭제 결재 차수';

CREATE INDEX IX_STOCATCH_APP_DOC ON STOCATCH (GATE_CD, APP_DOC_NUM, APP_DOC_CNT);
```

## 3. 상세 비즈니스/배치 동작 흐름

### A. 삭제 대상 선정 및 결재 요청
1. 운영자가 **삭제 대상 파일 관리** 화면(`fileDeleteList.jsp`)에 접근합니다.
2. 시스템은 등록일(`REG_DATE`) 기준으로 5년 이상 경과되었으며, 아직 삭제되지 않았거나 이전 결재가 반려/취소된 파일들을 자동 조회합니다.
3. 삭제하고자 하는 파일을 다중 선택하여 **[삭제 결재 상신]** 버튼을 누릅니다.
4. 시스템은 `docNumService`를 통해 `APPDOC` 결재 번호를 신규 채번하고, `STOCATCN` 마스터 테이블에 결재대기(`P`) 상태로 입력한 뒤, 선택된 파일들의 `STOCATCH` 레코드에 결재 번호(`APP_DOC_NUM`, `APP_DOC_CNT`)를 마킹하여 다른 결재에 중복 상신되지 않도록 잠금 처리합니다.

### B. 결재 승인 / 반려 / 취소(복구)
1. 결재자가 결재를 **승인**하면, 해당 결재 건의 `SIGN_STATUS`가 `E`(승인)로 변경됩니다.
2. 결재자가 결재를 **반려**하면, `SIGN_STATUS`는 `R`로 변경되고, `STOCATCH`에 잠겨있던 해당 결재 번호가 `NULL`로 초기화되어 파일들이 다시 삭제 대상 조회 목록에 나타납니다.
3. 승인 완료된 결재건 중, **물리 파일 백업 배치 작업이 수행되기 전**이라면 운영자는 **파일 삭제 및 백업 처리 현황** 화면(`fileDeleteStatus.jsp`)에서 **[삭제 복구/취소]** 버튼을 클릭하여 결재를 취소(`C`)하고 파일의 결재 정보를 회수할 수 있습니다.

### C. 시스템 배치 (물리 파일 백업 및 영구 삭제)
1. 매월 1회 실행되는 배치는 `runMonthlyBackupBatch`를 실행합니다.
2. 결재 승인(`E`) 완료된 파일 목록을 디비에서 가져와, Unix/Linux 호환 경로인 `upload/backup/YYYYMM/` 구조 하위로 물리 이동(Copy 후 원본 Delete)시킵니다.
3. 이동에 성공한 파일은 `STOCATCH.DEL_FLAG = '1'`로 업데이트하여 일반 조회 화면에서 영구 차단합니다.
4. 매주/매월 실행되는 영구 삭제 배치는 `runBackupPurgeBatch`를 호출하여 백업 경로 내 3개월(90일) 경과 파일을 완전히 디스크에서 영구 삭제합니다.
